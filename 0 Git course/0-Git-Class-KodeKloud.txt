GIT Merging branches

- once the feature has been fully tested, only then we can merge the feature branch into the master branch

- if we want to merge a branch we'd first need to checkout to the master branch if we are going to merge the feature/signup into the master branch

git checkout master

git merge feature/signup


Fast-forward vs. no-fast-forward!!!!
A fast-forward happens when the branch being merged is directly ahead of the branch you‚Äôre merging into ‚Äî that is, there‚Äôs a linear path from the current commit to the target commit (no divergent commits).
If no new commits have been added to main since branching off, merging feature into main can be done simply by moving the branch pointer forward:
git checkout main
git merge feature

No merge commit created ‚Äî Git just updates main to point to commit E.

Key traits:

Linear history (no extra merge commit)

Happens when the base of feature equals the tip of main

Equivalent to: git merge --ff

A no-fast-forward occurs when both branches have diverged ‚Äî meaning both have new commits since the split point. Git needs to create a new merge commit to combine them.
Now, merging feature into main:
git checkout main
git merge feature

Merge commit (M) created ‚Äî it ties together both branch histories.

Initialize Remote Repositories <-------------------------------------------------

The most common:
Github
Gitlab
BitBucket

We get access to a Connection String, which is the location of the repository!

git remote add origin
(a name for the remote and can have multiple remotes)
- 'origin' becomes the alias, so we don't have to remember the entire connection string. 

Full command:
git remote add origin http://..../.../name.git

- display all remote repositories
git remote -v

- we can 'push' and 'fetch' data to and from the remote repo from our local drive.
